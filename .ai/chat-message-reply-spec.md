# Спецификация: ответы на сообщения в чате

Статус: готово к реализации.

Источник требований: запрос пользователя и два референсных скриншота с Telegram-подобным поведением.

## 1. Цель

Добавить в чат полноценный сценарий ответа на сообщение:

- пользователь может выбрать любое уже существующее сообщение как сообщение для ответа;
- выбранное сообщение отображается над полем ввода до отправки или отмены;
- отправленное сообщение содержит компактный блок цитирования;
- клик по блоку цитирования прокручивает чат к исходному сообщению, ставит его по центру видимой области и временно подсвечивает.

## 2. Область изменений

Функция затрагивает только экран `/chat` и текущую модель сообщений этого экрана.

Вне области изменений:

- редактирование и удаление сообщений остаются как есть, кроме добавления рядом действия ответа;
- логика загрузки изображений, доставки, пагинации, realtime и optimistic UI не меняет свои существующие базовые правила, а только расширяется поддержкой ответа на сообщение;
- отдельный режим тредов не вводится.

## 3. Обязательные продуктовые решения

Во избежание двусмысленности принимаются следующие решения:

- Ответить можно на любое отправленное сообщение, включая собственное и чужое.
- Действие ответа доступно только для уже существующих сообщений с постоянным `id`. Для локально-оптимистичных сообщений без подтверждения сервера действие ответа недоступно.
- Режим ответа и режим редактирования взаимоисключающие.
- При переходе в режим ответа уже набранный текст в композере сохраняется.
- При переходе в режим редактирования активный ответ сбрасывается без подтверждения.
- При выборе нового сообщения для ответа предыдущее сообщение для ответа заменяется новым.
- Превью ответа, встроенное в отправленное сообщение, является снимком на момент отправки и не меняется после последующего редактирования исходного сообщения.
- Если исходное сообщение позже удалено, превью ответа в уже отправленном сообщении сохраняется, но переход по нему больше не выполняется.

## 4. Доступность действия "Ответить"

### 4.1 Desktop

- Одинарный клик по пузырьку сообщения открывает блок действий для этого сообщения.
- Для собственного сообщения блок действий содержит три кнопки в таком порядке: `Ответить`, `Редактировать`, `Удалить`.
- Для чужого сообщения блок действий содержит одну кнопку: `Ответить`.
- Повторный клик по тому же сообщению закрывает блок действий.
- Клик вне блока действий закрывает его.

### 4.2 Touch

- Долгое нажатие на пузырёк сообщения длительностью `450 ms` открывает контекстное меню действий.
- Для собственного сообщения меню содержит пункты в таком порядке: `Ответить`, `Редактировать`, `Удалить`.
- Для чужого сообщения меню содержит единственный пункт: `Ответить`.
- Отпускание пальца до истечения `450 ms`, движение пальца или отмена pointer-события не открывают меню.
- Тап вне меню закрывает меню.

### 4.3 Подписи и доступность

- Кнопка ответа имеет `aria-label`: `Ответить на сообщение`.
- Текст пункта меню на touch: `Ответить`.
- На desktop используется иконка ответа с hit-area `32x32 px`.

## 5. Состояние ответа в композере

### 5.1 Общий принцип

После нажатия `Ответить` над полем ввода появляется закреплённая панель ответа.

### 5.2 Структура панели ответа

Панель состоит из трёх зон:

- слева вертикальная полоса шириной `3 px`;
- по центру текстовое содержимое;
- справа кнопка закрытия.

### 5.3 Текстовое содержимое панели ответа

Отображаются две строки:

- строка 1: точный текст `В ответ {senderName}`;
- строка 2: превью исходного сообщения.

Правила формирования второй строки:

- если у исходного сообщения есть текст, показывается первая строка текста без переносов;
- длина превью ограничивается `80` символами;
- если текст длиннее, после `80` символов добавляется многоточие;
- если в исходном сообщении нет текста, но есть изображение, показывается точный текст `Фото`;
- если в исходном сообщении есть и текст, и изображение, показывается текстовое превью по общему правилу без добавления отдельной метки про изображение.

### 5.4 Цвета панели ответа

- Вертикальная полоса окрашивается в цвет ника автора исходного сообщения.
- Цвет ника автора исходного сообщения вычисляется тем же детерминированным алгоритмом и из той же палитры, что уже используется для имени автора входящих сообщений в ленте.
- Текст строки `В ответ {senderName}` окрашивается в тот же цвет, что и полоса.

### 5.5 Закрытие панели ответа

Панель ответа исчезает в любом из следующих случаев:

- пользователь нажал крестик;
- пользователь успешно отправил сообщение;
- пользователь вошёл в режим редактирования сообщения;
- пользователь выбрал другое сообщение для ответа, и панель была заменена новой.

При закрытии панели ответа текст, уже введённый в textarea, не очищается.

## 6. Отправка сообщения с ответом

### 6.1 Что отправляется

При активном ответе новое сообщение отправляется с привязкой к исходному сообщению и со снимком данных для отображения превью.

### 6.2 Состав данных ответа

В каждой записи ответа должны быть доступны следующие данные:

- идентификатор исходного сообщения;
- идентификатор автора исходного сообщения;
- отображаемое имя автора исходного сообщения;
- снимок превью исходного сообщения;
- признак того, было ли у исходного сообщения изображение;
- признак того, доступен ли переход к исходному сообщению.

### 6.3 Формирование снимка превью

Снимок превью создаётся в момент отправки нового сообщения и дальше не пересчитывается.

Правила:

- текст снимка берётся по тем же правилам, что и в панели ответа над композером;
- имя автора исходного сообщения сохраняется в том виде, в котором оно было показано в момент отправки;
- цвет полосы в ответе вычисляется не из сохранённого цвета, а заново по сохранённому `senderId` исходного сообщения, чтобы правило окраски было единым.

### 6.4 Ошибки и optimistic UI

- Optimistic-сообщение, созданное сразу после нажатия отправки, уже должно содержать блок ответа.
- Если отправка завершилась ошибкой, сообщение остаётся в списке в статусе ошибки вместе с блоком ответа.
- Пока отправка не завершилась успешно, панель ответа над композером сбрасывается сразу после постановки сообщения в очередь.

## 7. Отображение ответа в ленте

### 7.1 Расположение

Если сообщение является ответом, внутри его пузырька над основным текстом сообщения и над изображением отображается блок цитирования.

Порядок элементов внутри пузырька ответа:

- блок цитирования;
- изображение текущего сообщения, если оно есть;
- текст текущего сообщения, если он есть;
- строка времени и статуса.

### 7.2 Внешний вид блока цитирования

Блок цитирования имеет:

- левую вертикальную полосу шириной `3 px`;
- фон на один визуальный уровень светлее фона самого пузырька;
- скругление `10 px`;
- внутренние отступы `8 px`;
- нижний внешний отступ `6 px` до основного содержимого сообщения.

### 7.3 Содержимое блока цитирования

Блок цитирования всегда содержит две строки:

- строка 1: имя автора исходного сообщения;
- строка 2: превью исходного сообщения.

Правила строки 1:

- показывается полное отображаемое имя без сокращений;
- цвет текста совпадает с цветом полосы.

Правила строки 2:

- одна строка без переносов;
- максимум `80` символов;
- переполнение скрывается многоточием;
- если исходное сообщение было только с изображением, показывается `Фото`;
- если исходное сообщение было без текста и без изображения, показывается `Сообщение`.

### 7.4 Интерактивность

- Если у ответа есть доступный идентификатор исходного сообщения, весь блок цитирования кликабелен.
- Если переход недоступен, курсор указателя не показывается и клик ничего не делает.

## 8. Переход к исходному сообщению

### 8.1 Базовое поведение

По клику на блок цитирования выполняются действия строго в таком порядке:

1. Если исходное сообщение уже есть в DOM, начинается переход к нему.
2. Если исходного сообщения нет в DOM, клиент последовательно подгружает более ранние страницы истории, пока сообщение не будет найдено или пока история не закончится.
3. Если сообщение найдено, скролл доводит его до центра видимой области списка сообщений.
4. После завершения прокрутки к сообщению применяется временная подсветка.

### 8.2 Правила прокрутки

- Используется плавная прокрутка.
- Если у пользователя включено уменьшение анимации, прокрутка выполняется без анимации.
- Целевое сообщение должно оказаться по вертикальному центру видимой области scroll-контейнера с допустимым отклонением не более `24 px`.

### 8.3 Правила подсветки

- Подсветка длится `2000 ms`.
- Если пользователь повторно кликает по тому же блоку цитирования до окончания подсветки, таймер подсветки запускается заново.
- Подсветка применяется только к целевому пузырьку сообщения, а не ко всей строке с аватаром.
- Подсветка должна быть заметна и на входящих, и на исходящих сообщениях.

### 8.4 Если переход невозможен

Переход считается невозможным в одном из случаев:

- исходное сообщение было удалено, и у ответа больше нет валидной ссылки на него;
- вся доступная история подгружена, но сообщение не найдено.

В этом случае:

- прокрутка не выполняется;
- подсветка не выполняется;
- блок цитирования остаётся видимым как статичное превью;
- никаких toast-уведомлений не показывается.

## 9. Взаимодействие с режимом редактирования

Режим ответа и режим редактирования не могут существовать одновременно.

Точные правила:

- при запуске редактирования активный ответ немедленно сбрасывается;
- при запуске ответа активное редактирование немедленно закрывается без сохранения;
- текст, введённый в режиме обычного написания сообщения, не очищается при входе в режим ответа;
- текст в режиме редактирования не переносится в режим ответа.

## 10. Данные и контракт модели сообщения

### 10.1 Изменение модели сообщения в UI

Каждое сообщение в UI должно иметь необязательный вложенный объект `replyTo`.

Если объект `replyTo` отсутствует, сообщение не является ответом.

Если объект `replyTo` присутствует, в нём обязательно должны быть доступны:

- `messageId`: идентификатор исходного сообщения или `null`, если переход больше невозможен;
- `senderId`: идентификатор автора исходного сообщения;
- `senderName`: отображаемое имя автора исходного сообщения;
- `previewText`: строка превью по правилам этой спецификации;
- `hasImage`: был ли у исходного сообщения файл изображения;
- `isNavigable`: можно ли перейти к исходному сообщению по клику.

### 10.2 Изменение хранения в БД

В таблице сообщений должны храниться поля ответа:

- `reply_to_message_id`;
- `reply_to_sender_id`;
- `reply_to_sender_name`;
- `reply_to_preview_text`;
- `reply_to_has_image`.

Правила хранения:

- `reply_to_message_id` nullable;
- удаление исходного сообщения не удаляет ответы на него;
- при удалении исходного сообщения `reply_to_message_id` у существующих ответов сбрасывается в `null`;
- snapshot-поля ответа не меняются после создания сообщения.

### 10.3 Серверная нормализация

Сервер при создании сообщения-ответа обязан:

- проверить, что исходное сообщение существует, если его `id` передан;
- если исходное сообщение найдено, сохранить ссылку и snapshot-поля на основе серверных данных;
- если исходное сообщение не найдено к моменту отправки, создать сообщение без ссылки, но с snapshot-полями из клиентского состояния;
- вернуть в UI уже нормализованный объект `replyTo`.

## 11. Realtime, SSR и пагинация

### 11.1 SSR

Начальная загрузка страницы `/chat` должна получать поля ответа вместе с остальными полями сообщения.

### 11.2 Realtime INSERT и UPDATE

Новые и отредактированные сообщения, пришедшие по realtime, должны содержать объект `replyTo` в том же формате, что и SSR.

### 11.3 DELETE

При удалении исходного сообщения список сообщений не пересчитывает текст snapshot-превью в уже существующих ответах.

Поведение для уже загруженных ответов:

- `replyTo.messageId` становится `null`;
- `replyTo.isNavigable` становится `false`;
- `replyTo.senderName`, `replyTo.previewText`, `replyTo.hasImage` сохраняются.

### 11.4 Догрузка старых сообщений

Если пользователь кликает по цитате, а исходное сообщение ещё не загружено, система автоматически использует уже существующий механизм догрузки ранней истории до тех пор, пока:

- сообщение не будет найдено;
- либо не закончится история.

Во время такой догрузки:

- повторные клики по тому же блоку цитирования игнорируются;
- остальной чат остаётся интерактивным;
- отдельный индикатор загрузки для этого сценария не показывается.

## 12. Визуальные правила и размеры

Для единообразия принимаются следующие точные значения:

- ширина вертикальной полосы в панели ответа композера: `3 px`;
- ширина вертикальной полосы в блоке цитирования в сообщении: `3 px`;
- скругление блока цитирования: `10 px`;
- внутренние отступы блока цитирования: `8 px`;
- отступ между блоком цитирования и основным содержимым сообщения: `6 px`;
- максимальная длина превью везде: `80` символов;
- длительность подсветки найденного сообщения: `2000 ms`;
- длительность long-press на touch: `450 ms`.

## 13. Критерии приёмки

Функция считается реализованной корректно, только если выполняются все пункты ниже:

1. У любого подтверждённого сообщения доступно действие `Ответить`.
2. У собственного сообщения вместе доступны `Ответить`, `Редактировать`, `Удалить`.
3. У чужого сообщения доступно только `Ответить`.
4. После нажатия `Ответить` над композером показывается панель с текстом `В ответ {senderName}`, превью исходного сообщения и крестиком.
5. Нажатие на крестик закрывает панель ответа и не очищает введённый текст.
6. После отправки ответа новое сообщение в ленте содержит блок цитирования над своим основным содержимым.
7. Цвет полосы и имя автора в блоке цитирования совпадают по цветовой логике с ником автора исходного сообщения.
8. Превью исходного сообщения в ленте и над композером строится по одним и тем же правилам.
9. Клик по блоку цитирования прокручивает к исходному сообщению и ставит его по центру видимой области.
10. Найденное исходное сообщение подсвечивается на `2000 ms`.
11. Если исходное сообщение не загружено, история автоматически догружается до его нахождения или до исчерпания истории.
12. Если исходное сообщение удалено, превью в ответе остаётся, но переход по нему не выполняется.
13. Ответы корректно работают для текстовых сообщений, сообщений только с изображением и сообщений с текстом плюс изображение.
14. Optimistic-сообщение с ответом визуально совпадает по структуре с подтверждённым сообщением.
15. Realtime-вставка, SSR и догрузка старых страниц возвращают одинаковый формат данных ответа.

## 14. Что не допускается

Следующие варианты считаются несоответствием спецификации:

- исчезновение блока цитирования у старого ответа после удаления исходного сообщения;
- динамическое изменение текста snapshot-превью после редактирования исходного сообщения;
- очистка текста композера при отмене ответа;
- невозможность ответить на чужое сообщение;
- переход к исходному сообщению без его визуальной подсветки;
- прокрутка к неверному сообщению или к сообщению без попытки центрирования;
- наличие двусмысленности в том, что именно показывается как превью для сообщения без текста.
